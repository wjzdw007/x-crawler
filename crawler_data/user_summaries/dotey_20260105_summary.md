**新工具/技术**（如有）
- Claude Agent SDK：dotey 建议用它做第一个 Agent 原型，优点是快速起步、减少集成复杂度。
- Claude Code / Coding Agents：能把专家级 prompt 快速生成“玩具版”代码，适合原型验证与快速迭代。
- 多模型比较（如 Opus、Claude 4.5 等）：不同模型能力与成本差异显著，实践中要做对比选择。
- SDD（规格驱动开发）/“代码即文档”理念：在强模型下 SDD 的上限受限，直接以代码为准更实用但一致性难保。

**核心观点/方法论**（如有）
- 瓶颈从“实现”转向“表达/判断”：AI 降低实现成本，稀缺而昂贵的是能把问题、约束和 tradeoff 讲清楚的能力。
- 团队工作三件事：探索、验证、对齐——AI 主要替代的是“构建”这一步，前两步仍需人完成。
- 原型与生产分层：Vibe coding 做原型效率高，生产版必须考虑迁移成本、可维护性与统一架构。
- 个体/小团队窗口期：执行力被 AI 弥补，决策快、包袱轻的小团队反而更具竞争力。
- 下一个产品机会是“帮你想什么做”的工具：从执行工具向思考/探索工具迁移。

**实践经验/案例**（如有）
- Jaana Dogan 案例：在给定“存活的最佳想法”提示后，Claude 在小时级生成与团队一年成果相似的原型，说明 AI 放大已有认知而非凭空创造。
- 多分支/多模型实验：可并行生成不同实现以做比较，但代价是 token、时间与评审成本，不适合作为常态手段。
- 代码就是最可靠的 spec：文档常旧化，直接查看/运行代码更能反映真实实现，但要额外付出保持 spec 与实现一致的努力。
- 产品优化要看组织影响：过度或盲目优化（如极端提速）可能改变用户/组织行为，需谨慎评估影响与副作用。
- 企业转型建议：推行“软件优先”、让非技术职能拥抱 AI 工具，并把“做什么”作为战略核心。
