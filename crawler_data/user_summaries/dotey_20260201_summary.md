**新工具/技术**（如有）
- Codex — 用于反向工程：dotey 展示用 Codex 从已编译/混淆的 Electron 包中逐模块恢复 TypeScript 源码的实战流程，强调需配合外部记忆与自动化会话切换。  
- Nano Banana Pro（Gemini 变体）— 在图像生成/提示词调优上效果良好，适合复现高质量国风/3D 渲染风格的 prompt。  
- Claude Code / Agentic Search — 团队实践表明对代码检索，基于 agent 的关键字搜索在很多场景优于 RAG+向量库，且实现更简单、风险更小。  
- TiDB 的 non-blocking DDL — 推荐用于需要大量表/分支的场景（manus 使用案例），适配百万级表结构变更。  
- Filesystem-based tools（相对于 LSP）— 社区经验显示 LLM 更倾向并更好地配合基于文件系统的工具链。

**核心观点/方法论**（如有）
- 把 LLM 当作“有能力但需明确指令的队友”来用：清晰规则能避免模型走捷径（如只产出可编译的最小子集）。为什么重要：能得到更完整、可变更的产出。  
- 长期/跨会话任务必须有外部记忆与进度计划（PLAN 文件 + 自动续会脚本）。为什么重要：保持上下文连续性、避免重复说明和信息丢失。  
- 给模型可执行的验证回路（运行、测试）是高质量自动修复的关键。为什么重要：能让模型通过反馈自检并逐步收敛到正确实现。  
- 简单方案优先：Agentic keyword search 在工程复杂性、安全性、时效性上胜过 RAG 的场景并不少见。为什么重要：降低系统攻击面和信息陈旧风险。  
- 安全意识必须贯穿：平台暴露 API Key 会导致身份冒用与信息操控风险。为什么重要：影响名人/平台信誉与广泛滥用风险。

**实践经验/案例**（如有）
- 用 Codex 恢复源码（5 天）：步骤包括提取模块清单、设定“不求编译”的恢复规则、用 PLAN 文件保存进度、自动化新会话续接、并用 npm start + 集成测试驱动修复，最终得到可运行的源码。  
- Moltbook 泄露事件：实务建议立刻撤销/更换暴露的 API Key，平台用户在尝鲜时需留意数据与密钥泄露风险。  
- Prompt 泄露提醒：有推文指出“提示词就在 API 请求里面”，提示工程与敏感提示/样本可能被日志或数据库保存，需注意审计与脱敏。  
- Claude Code 的选择实践：早期用 RAG+向量库，后迁移到 agentic search，实际收益是系统更简单且减少安全/陈旧问题；可作为工程决策参考。
